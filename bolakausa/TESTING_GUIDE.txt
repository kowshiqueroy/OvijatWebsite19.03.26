This document serves as the comprehensive testing guide for the Bolakausa B2B

user credentials, database reset instructions, and step-by-step test flows.

--------------------------------------------------------------------------------
1. SYSTEM OVERVIEW & ARCHITECTURE
--------------------------------------------------------------------------------
Bolakausa is a B2B Wholesale e-commerce platform designed for wholesale customers
(e.g., delis, restaurants, caterers) to purchase products in bulk. The system
incorporates complex B2B features including tiered bulk pricing, dynamic delivery
fees and tax zones, order fulfillment lot tracking (FIFO), user verification, and
a direct client-staff chat system with attachments.

Technology Stack:
- Core Backend: PHP 7.4+ (Pure Vanilla PDO, Session-based auth)
- Database: MySQL/MariaDB (Innodb Engine, Foreign Key constraints)
- Frontend: HTML5, CSS3 (Modern Glassmorphism UI layout, FontAwesome icons)
- Routing: Front Controller (`index.php`) mapping clean request routes.

Base URL:
- Local Host: http://localhost/bolakausa/

Database Setup / Config File:
- config/database.php

5. KEY FUNCTIONALITIES & BUSINESS RULES
--------------------------------------------------------------------------------

1. Tiered Bulk Pricing:
   Product units prices drop dynamically when minimum order quantities are met.
   Example (Roma Tomatoes Case):
   - 1-4 cases: $18.99/ea
   - 5-9 cases: $17.50/ea
   - 10-19 cases: $16.75/ea
   - 20+ cases: $15.99/ea

2. Dynamic Delivery Matrix (Locations):
   Taxes and shipping charges are derived from the user's selected address location.
   Each location defines:
   - Tax Percent
   - Base Delivery Charge
   - Per Unit Weight Charge
   - Minimum and Maximum order amounts
   - Shipping Types (default, free, manual)

3. Diverse Payment Methods:
   - Cash on Delivery (COD)
   - Bank Transfer: Requires inputting a reference transaction number. The payment
     must be verified/approved by an Admin before fulfillment begins.
   - Pay Later (Credit Limit): Net 30 terms. Requires system settings allowance.
   - Stripe: Interactive mock Stripe card checkout (test number: 4242 4242 4242 4242).
   - Internal Wallet: Deducts directly from user's credit balance.

4. Wallet System & Top-up Request Flow:
   Users can top up their wallet balance using simulated Stripe or Bank Transfer.
   Stripe instantly creates a pending request. Bank Transfer allows attaching a proof
   file (image/pdf). Admins must approve the requests in the staff portal.

5. Inventory Lot Allocation (FIFO):
   Stock items are cataloged in designated "Inventory Lots" with expirations,
   received dates, and shelf locations. When orders are processed, the system
   deducts stock from lots based on FIFO (First-In, First-Out) rules.

6. Direct B2B Chat System:
   Allows clients to text staff directly. Support for attachments (e.g., invoices,
   receipt proofs) is integrated. Staff can view active client threads.
